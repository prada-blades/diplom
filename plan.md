# План серверной части ВКР: система бронирования общих корпоративных ресурсов

## Краткое резюме

- `Go` как основной язык backend-а.
- `REST API` как интерфейс для мобильного клиента.
- `PostgreSQL` как целевая основная БД.
- `JWT`-аутентификация.
- Слоистая архитектура: `HTTP/API -> service -> repository`.
- Развёртывание в виде внутреннего `on-prem` сервиса.
- Первая версия предметной области: `переговорные` и `рабочие места`.

## Обоснованный стек

- Язык: `Go`
- HTTP-слой: `net/http` или `chi`
- Хранение данных: `PostgreSQL`
- Миграции БД: `golang-migrate`
- Конфигурация: `.env` + переменные окружения
- Аутентификация: `JWT`
- Логирование: `log/slog`
- Документация API: `OpenAPI/Swagger`
- Контейнеризация: `Docker` + `docker-compose`
- Тестирование: встроенный `testing` + интеграционные тесты

## Что включить в первую реализацию

1. Управление пользователями и ролями.
2. Справочник ресурсов: переговорные и рабочие места.
3. Просмотр доступности ресурсов по интервалу.
4. Создание бронирования.
5. Отмена бронирования.
6. Просмотр своих бронирований.
7. Администрирование ресурсов.
8. Базовая отчётность по загрузке ресурсов.

## Роли

- `employee`: ищет ресурс, создаёт и отменяет свои брони.
- `admin`: управляет ресурсами, видит все брони, может отменять записи.

## Ключевые бизнес-правила

- Нельзя создать две пересекающиеся брони на один ресурс.
- У бронирования есть статус: `active`, `cancelled`.
- Нельзя бронировать на прошедшее время.
- Можно ограничивать максимальную длительность брони.
- Отмена доступна владельцу брони и администратору.

## Архитектура реализации

- `cmd/api` или `main.go`: точка входа
- `internal/config`: конфигурация
- `internal/http`: роуты, handlers, middleware
- `internal/service`: бизнес-логика
- `internal/repository`: работа с данными
- `internal/domain`: сущности
- `internal/auth`: аутентификация
- `migrations`: SQL-миграции
- `docs`: описание API и схем

Архитектурный стиль:

- монолитный backend;
- модульная внутренняя структура;
- без микросервисов на первом этапе.

## Основные сущности

### User

- `id`
- `full_name`
- `email`
- `password_hash`
- `role`
- `created_at`

### Resource

- `id`
- `name`
- `type` (`meeting_room`, `workspace`)
- `location`
- `capacity`
- `description`
- `is_active`

### Booking

- `id`
- `resource_id`
- `user_id`
- `start_time`
- `end_time`
- `status`
- `purpose`
- `created_at`

## REST API первой версии

- `POST /auth/register`
- `POST /auth/login`
- `GET /me`
- `GET /resources`
- `GET /resources/{id}`
- `POST /resources` (admin)
- `PUT /resources/{id}` (admin)
- `DELETE /resources/{id}` (admin, мягкое отключение)
- `GET /availability?start=...&end=...&type=...`
- `GET /bookings/my`
- `POST /bookings`
- `DELETE /bookings/{id}`
- `GET /admin/bookings`
- `GET /admin/reports/utilization`

3. Создание ресурса администратором.

## Допущения

- Реализация делается как `учебный MVP`.
- Основной контур внедрения: `on-prem`.
- Первая версия покрывает только `переговорные` и `рабочие места`.
- Интеграция с `LDAP/AD`, почтой и календарями пока не включается.
- Уведомления и внешние интеграции остаются как вторую очередь.
